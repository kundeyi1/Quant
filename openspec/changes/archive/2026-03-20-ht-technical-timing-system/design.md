## Context

当前代码库中，择时逻辑（之前称为 `filters`）散布在 `filters/signal_filters.py` 中，且以 `SignalFilters` 类的静态方法形式存在。华泰择时体系的引入需要计算多维度的指标（RSRS、均线、成交量等），这些指标不仅需要指数价格数据，还需要成交量等扩展数据。现有 `DataManager` 已具备读取 OHLCV 的基础，但 `filters` 的类封装结构不利于平铺化的函数库管理。

## Goals / Non-Goals

**Goals:**
- **平铺化函数库**：在 `timing/market_timing.py` 中实现独立的择时函数，遵循 `ht_<indicator>` 和 `gx_pit_<indicator>` 命名规范。
- **灵活输出规范**：原本离散的过滤器（gx_pit）继续输出 0/1；华泰系列函数按照研报 pdf （D:\Dev\研报\2025-12-28_华泰证券_金融工程_a股择时之技术打分体系.pdf）实现原始得分/连续信号输出。
- **解耦类封装**：移除 `SignalFilters` 类，采用顶层函数 (Top-level Functions) 模式。
- **算子复用**：深度集成 `core/NumericalOperators` 进行 RSRS 等回归计算。

**Non-Goals:**
- **不改变选股逻辑**：本次变更仅涉及“市场择时”（是否入场），不修改选股因子的具体实现。
- **不开发 UI/可视化**：择时曲线的绘图逻辑维持现状，不在本次设计范围内。
- **不引入外部 API**：所有数据计算均基于已读取的指数宽表，不实时爬取。

## Decisions

### 1. 结构变更：从 Class 到 Module
- **选择**：放弃类封装，使用模块内的顶层函数。
- **理由**：择时函数通常是无状态的纯函数（Stateless Pure Functions），使用类封装会增加 `import` 的层级负担，且无法直接作为参数传递。
- **备选**：维持类结构但重命名。弃用理由：不够灵活，且难以体现平铺化的函数库理念。

### 2. 命名约定：语义化前缀
- **选择**：`ht_`（指代华泰专有的算法）、`gx_pit_`（原有逻辑）、以及**无前缀**的通用数学命名。
- **理由**：函数名称应表达其业务或数学含义（如 `ht_trend_regime` 代表华泰定义的趋势择时），而不应将具体参数（如窗口大小 5/60）写死在函数名中。具体的参数应通过默认参数（default arguments）实现，以保持接口的语义化。

### 3. 处理连续信号：按研报原样输出
- **选择**：华泰系列择时函数返回连续的分数或指标值（如 RSRS 的 Z-Score），暂不执行 0/1 划分。
- **理由**：保留原始颗粒度，便于后续在回测时动态调整阈值或进行多因子融合。
- **备选**：函数内部强制离散化。弃用理由：当前阶段需要先复现研报效果，过早离散化会丢失信息。

### 4. 数据输入：标准化 OHLCV
- **选择**：所有子函数内部调用通用辅助函数 `_standardize_input`。
- **理由**：华泰指标（如成交量、均线）对输入列名（`close`, `volume` 等）有强依赖，通过辅助函数确保输入 DataFrame 的稳健性。

## Risks / Trade-offs

- **[Risk] 旧代码兼容性** → [Mitigation] 暂时保留 `filters/` 目录中的旧文件，或在 `timing/__init__.py` 中提供后向兼容的映射，直到所有测试脚本更新完毕。
- **[Risk] RSRS 计算效率** → [Mitigation] 使用 `NumericalOperators` 中的线性代数学算子（Numpy 底层），避免 Pandas 循环。
- **[Risk] 指数数据不完整** → [Mitigation] 如果输入数据缺少 `volume` 列，华泰相关函数应抛出明确的错误或返回全 1 以防漏报。
